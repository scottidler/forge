# Design Document: Forge -- MWP Pipeline Runner

**Author:** Scott Idler
**Date:** 2026-03-12
**Status:** In Review
**Review Passes Completed:** 5/5

## Summary

Forge is a Rust CLI tool that implements the Model Workspace Protocol (MWP) as a portable "briefcase" pattern. It defines content pipelines in YAML, deploys working scaffolding into any directory via symlinks (`forge unpack`), runs staged AI-assisted workflows using fabric patterns, and retracts cleanly when done (`forge pack`). Pipeline state is tracked via TaskStore; finished artifacts land in configured destinations (Obsidian vault, `docs/design/`, Confluence).

**Where forge fits in the tool ecosystem:**

| Tool | Domain | Execution Model |
|------|--------|-----------------|
| **Otto** | Build/CI tasks | Stateless DAG -- run tasks, done |
| **obsidian-borg** | Content ingestion (URL -> vault) | Stateless atomic -- one URL in, one note out |
| **Forge** | Content pipelines (research, drafts, specs) | Stateful staged -- unpack, run stages with review, pack |
| **Loopr** | Multi-agent code coordination | Concurrent branching -- multiple agents, git worktrees |

## Problem Statement

### Background

Scott Idler manages 377+ repositories across personal (`scottidler`) and work (`tatari-tv`) GitHub organizations. His work involves producing structured content -- design docs, tech specs, meeting notes, status updates, Jira epics -- alongside code. Jake Van Clief's Model Workspace Protocol (MWP) provides a proven framework for staged, AI-assisted content pipelines with human review gates. However, MWP's reference implementation uses a static folder hierarchy that assumes a creator camps in one workspace. Scott's workflow is the opposite: he moves between repos constantly, works primarily in the terminal and neovim, and cannot tolerate folder sprawl across hundreds of repos.

Scott already has the building blocks for a better approach:

- **TaskStore** -- a JSONL + SQLite database with git-native merge drivers, perfect for pipeline state
- **Otto** -- a YAML-driven task runner that proves config-driven pipelines work
- **obsidian-borg** -- a Rust CLI that already implements an MWP-like pipeline (normalize -> fetch -> summarize -> tag -> route -> render) for content ingestion
- **Fabric** -- a CLI that wraps LLM calls in reusable "patterns" for structured text processing
- **Scaffold** -- a Rust CLI project generator that provides conventions for config, logging, and CLI structure

### Problem

There is no tool that:

1. Defines reusable content pipelines in YAML (stages, fabric patterns, references, output targets)
2. Unpacks pipeline working state into any directory without permanent pollution
3. Tracks pipeline runs, stages, and artifacts across the filesystem
4. Separates pipeline definitions (factory) from working artifacts (intermediates) from finished products (finals)
5. Integrates with fabric for AI-powered stage execution

The alternatives are: manually running fabric commands and managing files ad-hoc, or adopting MWP's folder hierarchy which doesn't fit a multi-repo terminal workflow.

### Goals

- Define content pipelines declaratively in YAML with stages, fabric patterns, references, and output routing
- Implement the briefcase pattern: `forge unpack` deploys `.forge/` into the current directory; `forge pack` retracts it
- Track pipeline state (run ID, current stage, artifact paths, timestamps) via TaskStore
- Store intermediate artifacts as real Markdown files in a managed directory (`~/.local/share/forge/runs/<run-id>/`)
- Route final outputs to configured destinations (Obsidian vault paths, `docs/design/`, Confluence)
- Provide CLI commands for pipeline lifecycle: `unpack`, `pack`, `run`, `ls`, `show`, `history`
- Integrate fabric as the primary stage executor
- Support human review gates between stages

### Non-Goals

- **Real-time multi-agent coordination** -- that's loopr's domain
- **Replacing obsidian-borg** -- borg handles high-volume automated ingestion; forge handles deliberate, staged content creation
- **GUI or IDE integration** -- terminal-first, neovim-compatible
- **Automatic stage advancement** -- all commands are manual; automation comes later
- **Confluence/Jira API integration in v1** -- final output lands as files; publishing is a future concern
- **Meeting notes pipeline in v1** -- blocked on transcript access (Gemini/Granola dependency)

### Data Model

#### Pipeline Definition (YAML)

```yaml
# ~/repos/scottidler/forge/pipelines/techspec.yml
# Top-level keys ARE the pipeline fields (no wrapper key needed -- the file IS the pipeline)
name: techspec
description: "Research, outline, draft, and review a technical specification"

output:
  destination: "docs/design/"  # relative to working dir where unpack was run
  filename: "{date}-{slug}.md" # {date} = YYYY-MM-DD, {slug} = from --input or --slug flag
                                # e.g., forge unpack techspec --slug "state-recovery"
                                # produces: docs/design/2026-03-12-state-recovery.md

references:                      # pipeline-level refs available to all stages
  - references/voice.md
  - references/templates/techspec.md

stages:
  - name: research
    description: "Gather context, explore prior art, identify constraints"
    pattern: extract_article_wisdom  # fabric pattern name
    references:                      # stage-specific refs (merged with pipeline-level)
      - references/rubrics/research-rubric.md
    review: false

  - name: outline
    description: "Create structural outline with section headers and key points"
    pattern: create_outline
    references:
      - references/templates/techspec.md
    review: true  # human review gate -- forge run stops here until user continues

  - name: draft
    description: "Write full draft following outline and voice"
    pattern: write_document
    references:
      - references/voice.md
    review: true

  - name: review
    description: "Self-review against rubric, produce final"
    pattern: review_against_rubric
    references:
      - references/rubrics/techspec-rubric.md
    review: true
```

#### forge.yml (Top-Level Config)

```yaml
# ~/repos/scottidler/forge/forge.yml
forge:
  version: "1"
  home: ~/repos/scottidler/forge  # forge repo root (for resolving pipeline + reference paths)
  store: ~/.local/share/forge     # artifact store root (shellexpand handles ~/)
  # TaskStore lives inside the artifact store: ~/.local/share/forge/.taskstore/
  # This co-locates pipeline state with artifacts. Store::open() creates the
  # .taskstore/ directory structure (JSONL files + SQLite index) automatically.

  # Pipeline definitions -- keys are names, values are paths to YAML files
  # relative to the forge repo root
  pipelines:
    techspec: pipelines/techspec.yml
    research: pipelines/research.yml
    confluence-doc: pipelines/confluence-doc.yml
    status-update: pipelines/status-update.yml
    jira-epic: pipelines/jira-epic.yml

  # Fabric configuration
  fabric:
    binary: fabric
    model: ""  # empty = use fabric's default model

  # Default references available to all pipelines (merged with per-pipeline refs)
  global_references:
    - references/voice.md
```

#### TaskStore Record: PipelineRun

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct PipelineRun {
    pub id: String,                    // UUIDv7
    pub pipeline: String,             // "techspec", "research", etc.
    pub working_dir: String,          // where unpack was run (e.g. ~/repos/scottidler/loopr)
    pub status: RunStatus,            // Unfurled, InProgress, Packed, Completed, Abandoned
    pub current_stage: usize,         // 0-indexed stage number
    pub created_at: i64,              // millis since epoch
    pub updated_at: i64,              // millis since epoch
    pub input: Option<String>,        // initial input (file path, URL, topic description)
    pub stages: Vec<StageRecord>,     // per-stage state
    pub final_destination: Option<String>,  // where the final output was written
}

#[derive(Debug, Serialize, Deserialize)]
pub struct StageRecord {
    pub name: String,
    pub status: StageStatus,          // Pending, InProgress, Review, Completed, Skipped
    pub artifact_path: Option<String>, // path to output file in artifact store
    pub started_at: Option<i64>,
    pub completed_at: Option<i64>,
}

#[derive(Debug, Serialize, Deserialize)]
pub enum RunStatus {
    Unpacked,    // .forge/ deployed, no stages run yet
    InProgress,  // at least one stage started
    Packed,      // .forge/ retracted, intermediates archived
    Completed,   // final output delivered
    Abandoned,   // packed without completing
}

#[derive(Debug, Serialize, Deserialize)]
pub enum StageStatus {
    Pending,
    InProgress,
    Review,      // waiting for human review
    Completed,
    Skipped,
}

// TaskStore Record trait implementation
impl Record for PipelineRun {
    fn id(&self) -> &str { &self.id }
    fn updated_at(&self) -> i64 { self.updated_at }
    fn collection_name() -> &'static str { "pipeline_runs" }

    fn indexed_fields(&self) -> HashMap<String, IndexValue> {
        let mut fields = HashMap::new();
        fields.insert("pipeline".into(), IndexValue::String(self.pipeline.clone()));
        fields.insert("working_dir".into(), IndexValue::String(self.working_dir.clone()));
        fields.insert("status".into(), IndexValue::String(format!("{:?}", self.status)));
        fields.insert("current_stage".into(), IndexValue::Int(self.current_stage as i64));
        fields
    }
}

// This enables queries like:
//   store.list::<PipelineRun>(&[Filter { field: "status", op: Eq, value: String("Unpacked") }])
//   store.list::<PipelineRun>(&[Filter { field: "pipeline", op: Eq, value: String("techspec") }])
//   store.list::<PipelineRun>(&[Filter { field: "working_dir", op: Eq, value: String("/home/...") }])
```

### Stage Execution Contract

This is the core of forge -- how stages actually run. Each stage follows this contract:

**Input composition:** When `forge run` executes stage N, it builds the input for fabric by concatenating:

1. The **stage description** from the pipeline YAML (as a system-level instruction)
2. The **previous stage output** (stage N-1's `.forge/<NN>-<name>.md`), or the `--input` file/text for stage 1
3. The **reference material** -- contents of all reference files (pipeline-level globals + stage-specific), read from the symlinks in `.forge/references/`

These are concatenated with clear delimiters and piped to fabric's stdin:

```
--- TASK ---
{stage.description}

--- PREVIOUS OUTPUT ---
{contents of previous stage .md, or initial input}

--- REFERENCE: voice.md ---
{contents of voice.md}

--- REFERENCE: techspec.md ---
{contents of techspec template}
```

Fabric then processes this with the named pattern: `fabric -p {stage.pattern}`.

The output from fabric's stdout is written to `.forge/<NN>-<name>.md`.

**Reference path resolution:** All reference paths in pipeline YAML are relative to the forge repo root. The forge repo root is determined by:

1. `FORGE_HOME` environment variable (if set)
2. The `home` field in `~/.config/forge/forge.yml`
3. Auto-detection: walk up from the `forge` binary's location looking for `forge.yml`

This means `references/voice.md` resolves to `$FORGE_HOME/references/voice.md`.

**Review gates:** When a stage has `review: true`, after writing the output file, forge:

1. Prints the output to stdout (so the user sees it in their terminal)
2. Sets the stage status to `Review` in TaskStore
3. Exits with code 0 (not an error -- the stage succeeded, it's waiting for the human)

The user reads the output (in terminal, or opens `.forge/<NN>-<name>.md` in neovim), optionally edits it, and then runs `forge run` again to advance. The next `forge run` call sees the current stage is in `Review` status, marks it `Completed`, and proceeds to execute the next stage.

**Manual edits:** If the user edits a stage output file in `.forge/` between runs, the edited version becomes the input for the next stage. Forge does not track or diff manual edits -- the file on disk is the source of truth for that stage's output.

### API Design (CLI Interface)

```
forge <COMMAND>

Commands:
  unpack <pipeline> [--input <FILE|URL|TEXT>] [--slug <NAME>]
      Deploy pipeline scaffolding into current directory.
      Creates .forge/, symlinks references, adds .forge to .gitignore.
      --slug sets the output filename slug (e.g., "state-recovery").
      Errors if .forge/ already exists (collision detection).

  pack [--abandon]
      Retract .forge/ from current directory.
      Archives intermediates to artifact store.
      Writes final output to configured destination (unless --abandon).
      Removes .forge/ and .gitignore entry.

  run [<stage-name>] [--input <FILE>]
      Execute the next pending stage (or a named stage) of the active pipeline.
      If the current stage is in Review status, calling `run` implicitly approves
      the review and advances to the next stage. This keeps the interface simple:
      the user runs stages forward one at a time, reviewing output between calls.
      Reads previous stage output + references, runs fabric pattern.
      Writes output to .forge/<NN>-<name>.md.
      If stage has review: true, prints output and sets status to Review.

  ls [--all]
      List active (unpacked) pipeline runs.
      --all includes packed/completed/abandoned runs.

  show [<run-id>]
      Show details of current or specified pipeline run.
      Displays pipeline name, working dir, stage status, artifact paths.

  history [<pipeline>] [--limit N]
      Show history of pipeline runs, optionally filtered by pipeline type.

  describe <pipeline> [--stage N]
      Print pipeline definition. With --stage, print just that stage's contract.

  refs <pipeline> [--stage N]
      List reference material for a pipeline or specific stage.

  pipelines
      List all available pipeline definitions.

Global Options:
  -c, --config <PATH>    Path to forge.yml config
  -v, --verbose          Enable verbose output
  --version              Print version
```

### Implementation Plan

#### Phase 1: Foundation

- Replace scaffold placeholder code with forge-specific config, CLI, and data model
- Define `forge.yml` schema and pipeline YAML schema
- Implement pipeline loading and validation
- Integrate TaskStore as a library dependency
- Define `PipelineRun` and `StageRecord` as TaskStore records
- Implement `forge pipelines` and `forge describe`

#### Phase 2: Briefcase Core

- Implement `forge unpack`:
  - Parse pipeline definition
  - Create `.forge/` directory
  - Symlink references from forge repo
  - Add `.forge` to `.gitignore` with a marker comment (`# forge-managed`);
    if `.forge` is already in `.gitignore` without the marker, skip (user owns it)
  - Create TaskStore record
  - Write `.run-id` file
- Implement `forge pack`:
  - Copy intermediates to artifact store (`~/.local/share/forge/runs/<id>/`)
  - Write final output to configured destination
  - Update TaskStore record
  - Remove `.forge/` directory
  - Remove `.forge` from `.gitignore` only if it has the `# forge-managed` marker

#### Phase 3: Stage Execution

- Implement `forge run`:
  - Determine current stage from TaskStore or `.forge/` state
  - Load previous stage output (or initial input for stage 1)
  - Load stage references (read symlinked files)
  - Execute fabric pattern with input + references as context
  - Write output to `.forge/<NN>-<name>.md`
  - Update TaskStore stage status
  - If review gate: display output, set status to Review
- Review gate logic: `forge run` on a pipeline where current stage is Review
  implicitly approves and advances before executing the next stage

#### Phase 4: Visibility and History

- Implement `forge ls` -- query TaskStore for active runs
- Implement `forge show` -- display run details
- Implement `forge history` -- query completed/abandoned runs
- Implement `forge refs` -- list reference material

#### Phase 5: Polish

- Error handling: graceful messages for missing fabric, missing patterns, broken symlinks
- Fabric availability check (reuse obsidian-borg pattern)
- Shell completions
- `forge.yml` validation with helpful error messages
- Write initial pipeline definitions: `techspec`, `research`, `confluence-doc`
- Write reference material: `voice.md`, templates, rubrics

## Alternatives Considered

### Alternative 1: MWP Folder Hierarchy (Jake's Original Approach)

- **Description:** Each pipeline is a directory tree with `stages/01_name/CONTEXT.md`, `references/`, `output/` subdirectories. The user navigates into stage folders to work.
- **Pros:** Simple mental model; no tooling required; matches the paper's reference implementation.
- **Cons:** Requires navigating deep directory trees in terminal; doesn't scale to 377+ repos; folder sprawl; assumes the user stays in one workspace.
- **Why not chosen:** Scott works across many repos, uses terminal + neovim, and explicitly rejected the directory crawling experience.

### Alternative 2: Workspace Folders Inside Each Repo

- **Description:** Each repo gets a `.workspace/` or `docs/workspace/` folder containing pipeline scaffolding.
- **Pros:** Pipeline context lives alongside the code it serves; no symlinks needed.
- **Cons:** 377+ repos would each get workspace folders; massive boilerplate; maintenance burden; clutters repos.
- **Why not chosen:** Scale makes this absurd. "Hopefully you see how asinine that option B suggestion was."

### Alternative 3: Extend Otto with Pipeline Semantics

- **Description:** Add stage chaining, reference loading, and fabric integration to otto's task runner.
- **Pros:** Reuses existing tool; no new binary; already has YAML config and task DAG execution.
- **Cons:** Otto is a build/CI task runner -- conflating it with content pipelines muddies its purpose; otto tasks are stateless (run and done), pipelines are stateful (pause, review, resume); different execution model.
- **Why not chosen:** Clean separation of concerns. Otto runs tasks. Forge runs pipelines. They can coexist (and forge's CI runs through otto).

### Alternative 4: Extend obsidian-borg

- **Description:** Add pipeline subcommands to obsidian-borg beyond content ingestion.
- **Pros:** Already has fabric integration, config loading, vault output.
- **Cons:** obsidian-borg is purpose-built for URL → Obsidian note ingestion; adding general-purpose pipelines would dilute its focus; different lifecycle (borg is stateless/atomic, forge is stateful/staged).
- **Why not chosen:** obsidian-borg does one thing well. Forge does a different thing. They share patterns (fabric integration, config conventions) but are separate tools.

### Alternative 5: Git Worktrees as Pipeline State

- **Description:** Use git worktrees to isolate pipeline working state.
- **Pros:** Git-native isolation; familiar tooling; Git Butler support.
- **Cons:** Worktrees are execution environments, not databases; `git worktree list` would show dozens of abandoned pipeline runs; no query capability; metaphor doesn't fit.
- **Why not chosen:** A worktree is an isolated checkout, not a state store. TaskStore is purpose-built for this.

## Technical Considerations

### Dependencies

**Internal (Scott's tools):**

| Dependency | Usage | Integration |
|------------|-------|-------------|
| TaskStore | Pipeline state tracking | Cargo library dependency (`taskstore = { path = "..." }` or published crate) |
| Fabric | Stage execution | Shell command (`fabric -p <pattern>`) |
| Otto | CI/CD for forge itself | `.otto.yml` (no code dependency) |
| Scaffold | Initial project structure | Already applied (one-time) |

**External (Rust crates):**

| Crate | Purpose |
|-------|---------|
| clap 4 (derive) | CLI parsing |
| eyre | Error handling |
| serde + serde_yaml | YAML config/pipeline parsing |
| tokio | Async runtime (for fabric subprocess execution) |
| uuid (v7) | Run IDs (time-sortable) |
| chrono | Timestamps |
| dirs | Platform directories |
| colored | Terminal output |
| shellexpand | `~/` expansion in paths |
| log + env_logger | Logging |

**TaskStore integration approach:** The cleanest path is adding `taskstore` as a path dependency during development, publishing it to crates.io when stable, and switching to a version dependency. TaskStore's `Record` trait is generic -- `PipelineRun` implements it and gets JSONL persistence + SQLite indexing for free.

### Performance

- Pipeline operations are I/O-bound (reading YAML, creating symlinks, calling fabric)
- Fabric calls are the bottleneck -- each stage invokes an LLM via the fabric CLI
- TaskStore queries (SQLite) are sub-millisecond for the expected dataset size (hundreds of runs, not millions)
- `forge ls` should respond instantly; `forge run` response time is dominated by the LLM call

### Security

- **No secrets in pipeline definitions** -- patterns, templates, and references are plain text
- **Fabric API keys** -- managed by fabric's own config, not forge
- **Symlink safety** -- forge only creates symlinks pointing into the forge repo; never follows untrusted symlinks
- **`.gitignore` management** -- forge adds/removes `.forge` entries; never overwrites existing `.gitignore` content

### Testing Strategy

- **Unit tests:** Pipeline YAML parsing, config loading, path resolution, `.gitignore` manipulation, TaskStore record serialization
- **Integration tests:** Full `unpack` -> `run` -> `pack` cycle using a temp directory and mock fabric (a script that echoes formatted output)
- **Fabric mock:** A shell script at `tests/mock-fabric` that accepts `-p pattern` and returns canned output, avoiding LLM calls in tests
- **Coverage target:** 90%+ line coverage (Scott's convention)
- **CI:** `otto ci` runs lint + check + test (already configured in `.otto.yml`)

### Rollout Plan

1. **Phase 1 (Foundation):** Replace scaffold placeholders, define schemas, integrate TaskStore -- forge compiles and runs `forge pipelines` / `forge describe`
2. **Phase 2 (Briefcase):** Implement `unpack` and `pack` -- the core briefcase lifecycle works
3. **Phase 3 (Execution):** Implement `forge run` with fabric integration -- end-to-end pipeline execution
4. **Phase 4 (Visibility):** Implement `ls`, `show`, `history` -- full observability
5. **Phase 5 (Content):** Write pipeline definitions, templates, rubrics -- forge has useful pipelines to run
6. **Iterate:** Add pipelines as needs arise; refine based on real usage

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| TaskStore API changes break forge | Low | Medium | Pin TaskStore version; use path dep during co-development |
| Fabric patterns don't exist or produce poor output | Medium | Medium | Validate pattern existence at `unpack` time; allow shell commands as stage executors alongside fabric |
| `.gitignore` manipulation conflicts with user edits | Low | Low | Only append/remove the `.forge` line; never rewrite the file; use marker comments |
| Symlink targets move (forge repo path changes) | Low | Medium | Store forge repo path in config; validate symlinks at `unpack` time |
| Artifact store grows unbounded | Medium | Low | Add `forge gc` command to prune old runs; runs are small (markdown files) |
| Pipeline definitions are wrong/incomplete for real use | High | Medium | Start with one pipeline (techspec), iterate based on real usage before adding more |
| Stage output doesn't flow cleanly between stages | Medium | High | Define clear contract: each stage reads previous `.md` + references, writes next `.md`; test the contract in integration tests |
| Fabric not installed or pattern missing | Medium | High | Check fabric availability at `unpack` time (reuse obsidian-borg's `is_available()` pattern); validate pattern existence before first `run` |
| Machine crash leaves orphaned `.forge/` | Low | Medium | `forge ls` cross-references TaskStore records with filesystem; reports orphaned runs; `forge pack --abandon` cleans up |
| `.gitignore` doesn't exist in target directory | Low | Low | `forge unpack` creates `.gitignore` if needed; only adds the `.forge # forge-managed` line |
| Pipeline YAML changes while a run is active | Low | Low | Symlinks point to live files -- changes are picked up on next `forge run`. Document this as intentional (allows iteration). The pipeline definition at unpack time is not snapshotted. |

## Open Questions

- [x] ~~How should `forge run` compose the prompt for fabric?~~ Resolved: concatenate stage description + previous output + reference contents with delimiters, pipe to `fabric -p <pattern>` via stdin. See "Stage Execution Contract".
- [x] ~~What happens when the user edits a stage output manually?~~ Resolved: the edited file is the source of truth. Forge doesn't track edits -- the file on disk is what the next stage reads.
- [ ] Should `forge run` invoke fabric directly (subprocess) or should it also support an MCP-based execution path where Claude calls forge tools?
- [ ] Should the forge repo be the single source of truth for pipeline definitions, or should per-repo `forge.yml` overrides be supported?
- [ ] Should `forge pack` auto-detect the final stage output and copy it to the destination, or require the user to specify which file is the final?
- [ ] How should forge handle the case where the user wants to run a pipeline without being in a git repo (e.g., in `~/scratch/`)? The `.gitignore` management would need to be conditional.
- [ ] Should `forge unpack` accept a `--target <dir>` to deploy into a directory other than cwd?
- [ ] Should stage patterns support shell commands in addition to fabric pattern names? E.g., `command: "pdftotext -layout $input -"` as an alternative to `pattern:`.
