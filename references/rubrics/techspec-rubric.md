# Technical Specification Review Rubric

## Completeness
- [ ] All required sections are present
- [ ] Problem statement clearly articulates the "why"
- [ ] Proposed solution covers architecture, data model, and API
- [ ] At least 2 alternatives are considered with clear reasoning for rejection
- [ ] Implementation plan has distinct phases with deliverables
- [ ] Risks are identified with concrete mitigations

## Clarity
- [ ] A new team member could understand the proposal without prior context
- [ ] Technical terms are used consistently
- [ ] Diagrams support the text (not replace it)
- [ ] Trade-offs are explicitly stated, not implied

## Feasibility
- [ ] Dependencies are identified and their availability confirmed
- [ ] Performance characteristics are addressed for expected scale
- [ ] Security implications are considered
- [ ] Testing strategy covers unit, integration, and edge cases

## Actionability
- [ ] Implementation plan is specific enough to start coding from
- [ ] Open questions are clearly marked and don't block starting
- [ ] Success criteria are measurable
